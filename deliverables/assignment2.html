# The brief

This is the second of three take home assignments related to building a backend web service to support a simple application called tennisStar.

In the previous assignment you built the base for the tennisStar RESTful API using Node.js and the Express framework. For this assignment you will enhance that base application to use MongoDB for data persistance and add a second court resource.

# Core Requirements

  1. Using the Express.js framework, the API will expose a full set of CRUD routes (six, including both put and patch) for each of two resources: match, and court. All API resource paths must begin with /api. (ie /api/matches + /api/courts)

  2. Resource collections will be stored in MongoDB and accessed via Mongoose Model classes. The Model.schema for the resource objects will have the following properties.

# Court

A tennis court is the location where a match of tennis is played. Typically, this will be a park or tennis club, and there will be more than one actual court there.

For this API, we will want to know the name of the court, and the number of courts available.

Property Type Required Min Length Max Length
_id ObjectId true Auto generated by Mongodb
name String true 3 250
count Number true 1 99

# Match

Remember

The MongoDB driver will automatically assign the _id property.

  1. The court property of the match model should be an object id referencing the Court model. Whenever a match is fetched, the court property should be populated with all of the data for the corresponding court.

  2. Each resource should have its own Router module.

  3. Routes related to individual members of a resource collection should return a properly formatted error response with an error object for any invalid req.body or not found req.params.id value.

  4. Ensure that you write clean and readable code. Pay attention to:

  • no runtime errors
  • follow the project structure used in this court
  • properly formatted JSON responses with either data or error at the root
  • no debugging console logs
  • no commented out code
  • semantically descriptive names for variables and functions
  • use a .gitignore file and do not commit node_modules/ or .env files
  • include .env.example file
  • properly formatted package.json file
    • correct project name
    • your author details
    • start and dev scripts

  • Make sure to import and export correctly to and from each module

# Logistics

  • Start with the finished code in your assignment repo.
  • Create a branch called assignment-2.
  • If you are not happy with your assignment 1, feel free to copy my assignment 1 example into your assignment-2 branch as a starting point.
  • Migrate all of your service methods to use mongoose instead of an in memory matches array.
  • Build the project on your laptop.
  • Test each route with Postman.
  • Make git commits as you complete each requirement
  • When you are ready to submit, merge branch assignment-2 in the main branch, making sure both main AND assignment-2 are up to date in Github.
  • When everything is complete, push the final commit back up to GitHub and submit the GitHub repo's URL on Brightspace.

TIP

It is a good habit to make a git commit as you complete each logical requirement. This not only tracks your progress, but protects your working code in case you later break it and need to roll back.
