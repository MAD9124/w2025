# The brief

This is the third of three take home assignments related to building a backend web service to support a simple tennis application called tennisStar.

In the previous assignments you built the base for the tennisStar RESTful API using Node.js, the Express framework and MongoDB. For this assignment you will enhance that foundation with Google OAuth 2.0 authentication.

In addition to correctly implementing all of the requirements from the previous assignment. The application will implement the following.

# Core Requirements

  1. Create a new /auth router module that will support
  • /google | redirecting a user to google api for authentication
  • /google/callback | handle the callback from google and redirect to a given url with a JWT token in the query
    • ex http://localhost:3000/login-success?token=ey1290812ejn....
  1. The user schema should have the following properties:

# Court

A tennis court is the location where a match of tennis is played. Typically, this will be a park or tennis club, and there will be more than one actual court there.

For this API, we will want to know the name of the court, and the number of courts available.

Property Type Required Min Length Max Length
_id ObjectId true Auto generated by Mongodb
name String true 3 250
count Number true 1 99

# Match

Property Type Required Min Length Max Length
_id ObjectId true Auto generated by Mongodb
court Court true
player1 User true
player2 User true
sets [[Number, Number]] true 3 5
winner User true

Use document references

  • court should be stored as ObjectId in the db, referencing the Court collection
  • player1, player2 and winner should be stored as ObjectId in the db, referencing the User collection

  1. All requests to the match resource are only available to authenticated users. All requests to the court resource are available to all users, authenticated or unauthenticated.
  • An authenticated user will have a valid JWT token in the Authorization header, using the Bearer [token] strategy.
  1. All requests to the match resource should be limited to the authenticated users.
  • CREATE: when creating a match, player1 should automatically be the authenticated user. Only player2 needs to come from the input.
  • GET: normal behaviour, except a 401 should be thrown if authentication fails.
  • UPDATE / DELETE: a user can only update or delete a match that they created (ie player1 matches the authenticated user). A 403 should be thrown if an authenticated user tries to update or delete a match where they are not player1.
  • All responses from the match resource should include the populated Court (all properties) and the populated User (_id and name only, DO NOT INCLUDE googleId).

  1. Ensure all expected errors are properly formatted, and respond with the correct status code. This includes malformed object ids and failing mongoose validation errors.All errors messages should end with 2 periods

  2. Use security best practices to guard again Cross Site Scripting (XSS) and database injection attacks.

  3. Ensure that you write clean and readable code. Pay attention to:

  • no runtime errors
  • consistent 2 space indentation
  • USE YARN, not NPM
  • follow the project structure used in this court
  • properly formatted JSON responses with either data or error at the root
  • no debugging console logs
  • no commented out code
  • semantically descriptive names for variables and functions
  • use a .gitignore file and do not commit node_modules/ or .env files
  • include .env.example file
  • properly formatted package.json file
    • correct project name
    • your author details
    • start and dev scripts
